# SPEC-AI-008 — AI 프롬프트 레지스트리 + 슈퍼어드민 콘솔

> 마케팅(블로그) 프롬프트를 **코드 상수 → DB로** 이관하고, 슈퍼어드민 콘솔에서 CRUD·버전·활성화·**플레이그라운드(즉석 생성 미리보기)**로 튜닝한다.
> 교차 repo 기능: `ai`(prompt_override seam) · `api`(DB·CRUD·맥락주입·preview) · `web`(콘솔 UI).
> 근거: 마케팅 블로그 말투 품질을 "재배포 없이 반복 튜닝"하려는 운영 니즈. 참고 패턴: kikoai `prompts` 레지스트리(`/app/src/lib/prompts/registry.ts`, `052_prompts.sql`).

## 목표

블로그 생성 프롬프트(`geo_rules.py`의 `BLOG_SYSTEM`·`GEO_RULES`·`OUTPUT_SPEC`)와 모델·temperature를 DB에서 관리한다. 운영자는 슈퍼어드민 콘솔에서 프롬프트를 편집하고, **활성화 전에 샘플 입력으로 즉석 생성해 결과를 비교**한 뒤 활성 버전을 전환한다. ai-server는 stateless를 유지하며, 게이트웨이가 active 프롬프트를 로드해 요청에 주입한다. DB가 비어 있거나 장애여도 ai-server 코드 기본값으로 폴백해 출시 기능이 깨지지 않는다.

## 범위

### In
- **ai-server seam**: `POST /marketing/blog` 요청에 optional `prompt_override`(system_md/rules_md/output_spec_md/model/temperature) 추가. 있으면 사용, 없으면 `geo_rules.py` 기본값 폴백.
- **DB**: `ai_prompt` 테이블(채널+버전, active 1개, 본문 3조각, 모델·파라미터, 감사·soft delete).
- **게이트웨이**:
  - `AiPrompt` 엔티티·리포지토리.
  - 어드민 CRUD `/admin/prompts`(`@RequiresAdmin`): 목록·상세·생성·수정·활성화·삭제.
  - 활성화 = `@Transactional`(채널 내 기존 active=false → 새 active=true) + partial unique index 안전망.
  - `MarketingService.generateBlog`: active 프롬프트 로드(메모리 5분 캐시) → ai 호출에 `prompt_override` 주입.
  - 플레이그라운드 `POST /admin/prompts/preview`: 어드민이 보낸 임시 프롬프트 + 샘플 입력으로 ai 호출 → 결과만 반환(저장 X).
- **web 콘솔** `(console)/console/prompts`:
  - 목록(채널별·active 뱃지), 편집/신규(clone), 활성화/롤백.
  - 플레이그라운드 패널: 편집 중 프롬프트 + 샘플 입력으로 미리보기 생성.
- **seed**(선택): 현재 `geo_rules.py` 내용을 `blog` 채널 `v1 active`로 시드.

### Out (확장 seam만)
- 멀티채널(인스타·스레드) — `channel` CHECK에 자리만, v1은 `blog`.
- placeholder 치환 엔진({{TONE_SAMPLES}} 등) — v1은 동적 데이터(키워드·말투샘플·매장맥락)를 게이트웨이가 기존처럼 코드 주입. 정적 부분만 DB.
- 프롬프트별 A/B 자동 트래픽 분배 — 콘솔 수동 활성화로 대체.
- 비-마케팅 경로(chat/ocr/proactive)의 프롬프트 DB화 — 후속.

## 아키텍처 (4층 + seam)

```
[콘솔] 편집/활성화 ──> 게이트웨이 ai_prompt (DB)
                              │
[블로그 생성]                  ▼
web ─> 게이트웨이 MarketingService:
        1. tone_profile + store_context 조립 (기존)
        2. ai_prompt(channel='blog', active) 로드 (5분 캐시)
        3. ai-server 호출 시 prompt_override = {system_md, rules_md, output_spec_md, model, temperature}
                              │
                              ▼
ai-server /marketing/blog:
        override 있으면 사용, 없으면 geo_rules.py 폴백
        → blog.py: [system] + [rules + 동적데이터(키워드·말투샘플·매장맥락)] + output_spec 조립

[플레이그라운드] 콘솔 ─> 게이트웨이 /admin/prompts/preview ─(임시 프롬프트 override)─> ai-server ─> BlogDraft (저장 안 함)
```

**정적/동적 분리(핵심):** DB엔 정적 부분(시스템·규칙·출력스펙·모델·temp)만 저장. 동적 데이터(키워드·상황·메모·말투샘플·매장맥락)는 게이트웨이가 런타임에 주입(기존 유지).

## 데이터 모델 (게이트웨이 DB · 수동 DDL · FK 없음)

```sql
CREATE TABLE ai_prompt (
  id             BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  channel        VARCHAR(20) NOT NULL DEFAULT 'blog' CHECK (channel IN ('blog')),
  version        VARCHAR(40) NOT NULL,              -- 'v1', 'v2-말투강화', '2026-06-21-실험'
  is_active      BOOLEAN NOT NULL DEFAULT FALSE,
  system_md      TEXT NOT NULL,                     -- 페르소나 + 말투 모방 지시
  rules_md       TEXT NOT NULL DEFAULT '',          -- GEO 구조 규칙
  output_spec_md TEXT NOT NULL DEFAULT '',          -- 출력 JSON 스펙
  model          VARCHAR(64),                       -- 비우면 ai-server 기본
  temperature    NUMERIC(3,2),
  max_tokens     INTEGER,
  notes          TEXT,                              -- 변경 사유
  created_by     VARCHAR(120),                      -- 어드민 식별
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at     TIMESTAMPTZ,
  UNIQUE (channel, version)
);
CREATE UNIQUE INDEX uq_ai_prompt_active_per_channel
  ON ai_prompt(channel) WHERE is_active = TRUE AND deleted_at IS NULL;
CREATE TRIGGER update_ai_prompt_updated_at
  BEFORE UPDATE ON ai_prompt FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

- 채널당 active 1개만 강제(soft-delete 제외).
- 모델·temperature·max_tokens NULL이면 ai-server 기본값 사용.
- 토큰 사용량은 이 테이블에 기록하지 않음(생성 콘텐츠는 `ai_marketing_content` 소관).

## API 계약

### ai-server (snake_case, stateless)
`POST /marketing/blog` — 요청에 optional 필드 추가:
```jsonc
{
  // ...기존 필드(keyword, situation, memo, photo_urls, tone_samples, store_context)...
  "prompt_override": {            // optional. 없으면 geo_rules.py 기본값
    "system_md": "...",           // optional
    "rules_md": "...",            // optional
    "output_spec_md": "...",      // optional
    "model": "claude-sonnet-4-6", // optional, 있으면 요청 시 모델 빌드
    "temperature": 0.7            // optional
  }
}
```
- override의 각 텍스트 필드는 부분 적용(있는 것만 교체, 나머지는 기본값).
- override 텍스트는 시스템/구조 지시(신뢰 입력 — 어드민만 게이트웨이 통해 전달). 동적 사용자 데이터는 기존처럼 펜스 격리 유지.

### 게이트웨이 (camelCase, `@RequiresAdmin`, `/admin/prompts/*`)
| 엔드포인트 | 역할 |
|-----------|------|
| `GET /admin/prompts?channel=blog` | 목록(채널별, active 포함) |
| `GET /admin/prompts/{id}` | 상세(본문 전문) |
| `POST /admin/prompts` | 생성(clone 지원: 기존 id에서 복제). optional 즉시 활성화 |
| `PATCH /admin/prompts/{id}` | 수정(본문·모델·temp·notes·is_active). is_active=true 전환 시 활성화 트랜잭션 |
| `DELETE /admin/prompts/{id}` | soft delete. **active 버전은 삭제 거부**(다른 버전을 먼저 활성화해야) |
| `POST /admin/prompts/preview` | 플레이그라운드: `{promptDraft, sampleInput}` → ai 호출 → `{draft}` (저장 X) |

- 모델 화이트리스트 검증(허용 모델만). 텍스트 길이 상한.
- 활성화/수정/삭제 후 게이트웨이 프롬프트 캐시 무효화.

### web 콘솔 (`(console)/console/prompts`)
- `prompts/page.tsx`(목록) · `prompts/[id]/page.tsx`(편집+플레이그라운드) · `prompts/new/page.tsx`(신규/clone).
- 서버 액션 `lib/actions/admin-prompts.ts`, 타입 `types/admin-prompt.ts`.

## 활성화 · 캐시 · 폴백

| 항목 | 방식 |
|------|------|
| 활성화 원자성 | Spring `@Transactional`: 같은 channel 기존 active=false → 대상 active=true. partial unique index가 동시성 안전망 |
| 캐시 | 게이트웨이 메모리 5분(채널별). 활성화/수정/삭제 시 무효화 |
| 폴백 | `prompt_override` 없음/일부 누락/게이트웨이 로드 실패 → ai-server `geo_rules.py` 기본값. **DB 비어도 동작** |
| 롤백 | 콘솔에서 과거 버전 활성화 클릭 → 동일 트랜잭션 |

## 보안 (HARD)
- 콘솔 엔드포인트 전부 `@RequiresAdmin` 게이팅(기존 `AdminInterceptor`).
- `prompt_override`는 게이트웨이(어드민 인증 통과)만 ai-server에 전달 — 일반 사용자 경로(`/marketing/blog` via web)는 override 미포함. ai-server는 게이트웨이 신뢰(X-Internal-Key) 하에 override 수용.
- 동적 사용자 데이터(키워드·말투샘플)는 override와 무관하게 기존 `[USER INPUT — DATA ONLY]` 펜스 유지.
- 모델 화이트리스트·텍스트 길이 상한으로 남용/비용 폭발 방어.
- 플레이그라운드 호출도 LLM 비용 발생 → 콘솔에 모델·예상비용 표기, 어드민 한정.

## 의존성
- SPEC-AI-007(마케팅 블로그 생성 엔진) — 이 위에 prompt_override seam을 얹음.
- 게이트웨이 `kr.ai.flori.admin`(AdminInterceptor·RequiresAdmin·AdminWebConfig) 기존 게이팅.
- web `(console)` 슈퍼어드민 레이아웃·인증.

## 인수기준
1. `prompt_override` 없이 호출하면 기존과 동일(`geo_rules.py` 기본값)하게 동작한다(회귀 없음).
2. `prompt_override.system_md`만 보내면 시스템만 교체되고 rules/output_spec은 기본값을 쓴다(부분 적용).
3. `prompt_override.model`/`temperature`를 보내면 그 모델·온도로 생성된다.
4. 게이트웨이가 `ai_prompt` active 행을 로드해 생성 시 주입하고, active가 없으면 폴백한다.
5. 콘솔에서 새 버전 생성 → 활성화 시 같은 채널의 기존 active가 자동 비활성된다(채널당 active 1개 불변식).
6. 콘솔 플레이그라운드에서 임시 프롬프트 + 샘플 입력으로 미리보기가 생성되고, **DB·활성본에 영향을 주지 않는다**.
7. 비어 있는 `ai_prompt`(시드 없음) 상태에서도 블로그 생성이 정상 동작한다(폴백).
8. 콘솔/CRUD/preview는 비어드민 접근 시 차단된다(`@RequiresAdmin`).
9. 각 repo: `ruff`/`ktlint`/`eslint`+`tsc` 및 테스트 그린.
