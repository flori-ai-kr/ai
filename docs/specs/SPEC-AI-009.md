# SPEC-AI-009 — 비동기 블로그 생성 + 완료 알림(푸시 + 인앱)

> **상태: TODO (설계만 — 미구현).** SPEC-AI-008 세션 중 도출. 별도 브랜치/세션에서 구현 예정.
> 교차 repo: `api`(비동기 처리·상태·푸시·인앱 알림) · `web`(fire-and-forget UX·헤더 벨). ai-server는 변경 없음(생성 엔진 그대로).

## 배경 / 문제

블로그 초안 생성은 LLM 호출로 **15~40초** 걸린다. 현재(SPEC-007)는 **동기**라:
- web 서버액션(`generateBlogDraft`)이 전체 시간 블로킹 → 사용자가 화면에서 **무한 대기**.
- 페이지를 벗어나면 요청이 취소됨(결과 유실).

목표: 생성을 **백그라운드**로 돌려 사용자가 즉시 다른 일을 하게 하고, **완료 시 푸시 + 헤더 인앱 알림**으로 알린다.

## 목표

1. 블로그 생성을 비동기로: web은 "접수"만 받고 즉시 반환, 사용자는 나가도 됨.
2. 결과는 `ai_marketing_content`에 저장되어 "내가 만든 초안" 목록에 자동 노출(상태: 생성중→완료/실패).
3. 완료 시 **웹푸시**("블로그 초안이 완성됐어요") + **헤더 인앱 알림**(벨 뱃지 + 목록).

## 범위

### In
- **api 비동기 생성**: `POST /ai/marketing/blog`(또는 `/blog/async`)가 `ai_marketing_content`를 `status='generating'`로 즉시 생성하고 `contentId` 반환(202). Spring `@Async`로 백그라운드에서 ai-server 호출 → 성공 시 `status='done'`+draft 저장, 실패 시 `status='failed'`. 캡 검사는 접수 시점.
- **상태 컬럼**: `ai_marketing_content.status VARCHAR(20) DEFAULT 'done'`(기존 행 호환) + `error_message TEXT?`. 목록/상세 응답에 status 포함.
- **인앱 알림 피드(신규 범용)**: `notifications` 테이블(user_id, type, title, body, link, read_at, created_at) + 점주 API(`GET /notifications`, `GET /notifications/unread-count`, `POST /notifications/{id}/read`, `POST /notifications/read-all`). 생성 완료 시 1건 INSERT.
- **푸시 발송**: 완료 시 `PushService`/`PushDispatcher`로 해당 user_id 구독에 웹푸시(딥링크=마케팅 화면/해당 초안). 기존 `ReservationNotificationService` 패턴 재사용.
- **web fire-and-forget**: 제출 → "백그라운드 작성 시작" 토스트, 폼 비블로킹. 결과 미리보기는 폴링(또는 완료 알림 클릭)으로 갱신. 헤더 벨을 **범용 알림 피드**로 확장(기존 예약 리마인더와 통합 or 별도 탭).

### Out (이번 범위 밖)
- 실시간 푸시 외 전송(SSE/WebSocket 라이브 스트림) — 폴링으로 충분.
- 생성 취소/재시도 큐, 동시 생성 N개 제한(향후).
- 멀티 인스턴스 분산 작업 큐(현재 단일 인스턴스 가정 → `@Async` 스레드풀. 분산 필요 시 후속).

## 아키텍처

```
[제출] web → api POST /ai/marketing/blog (비동기)
                 │ 1) 캡 검사 + ai_marketing_content(status=generating) 저장 → contentId 즉시 반환(202)
                 │ 2) @Async: aiClient.generateBlog(...) (PromptResolver 주입 — SPEC-008)
                 ▼
        성공 → content.status=done + draft 저장 → notifications INSERT + PushDispatcher 발송
        실패 → content.status=failed + error_message → notifications INSERT(실패 안내)

[수신] web 헤더 벨: GET /notifications/unread-count(폴링) → 뱃지
       클릭 → GET /notifications → 목록 → 항목 클릭 시 link(해당 초안)로 이동 + read 처리
       (병행) 마케팅 화면이 열려 있으면 contentId 상태 폴링으로 미리보기 자동 채움
```

## 데이터 모델 (수동 DDL · FK 없음)

```sql
-- 1) 생성 상태 (기존 테이블 ALTER)
ALTER TABLE ai_marketing_content ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'done'
  CHECK (status IN ('generating','done','failed'));
ALTER TABLE ai_marketing_content ADD COLUMN error_message TEXT;

-- 2) 범용 인앱 알림
CREATE TABLE notifications (
  id         BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  user_id    BIGINT NOT NULL,                       -- 간접참조(테넌트 격리)
  type       VARCHAR(40) NOT NULL,                  -- 예: MARKETING_BLOG_DONE, MARKETING_BLOG_FAILED
  title      VARCHAR(200) NOT NULL,
  body       TEXT,
  link       VARCHAR(300),                          -- 클릭 시 이동 경로(예: /admin/marketing?content=123)
  read_at    TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_notifications_user ON notifications(user_id, created_at DESC) WHERE read_at IS NULL;
```

## 동시성 / 안전
- `@Async`는 별도 스레드 → 트랜잭션 분리. 백그라운드 메서드는 자체 `@Transactional`로 status 갱신.
- 멱등: 같은 contentId 중복 완료 처리 방지(상태가 generating일 때만 done 전환).
- 멀티테넌시: notifications·content 모두 user_id 격리(TenantIsolationGuard 통과 — UserId 메서드명).
- 캡: 접수 시점에 `AiUsageGuard` 검사(현행 유지).
- 푸시 실패는 생성 성공에 영향 없음(best-effort, 로깅).

## API 계약(초안)
| 엔드포인트 | 역할 |
|-----------|------|
| `POST /ai/marketing/blog` | 비동기 접수 → `{contentId, status:'generating'}` (202) |
| `GET /ai/marketing/contents/{id}` | 상세 + `status`(폴링으로 완료 확인) |
| `GET /notifications?unreadOnly=` | 인앱 알림 목록 |
| `GET /notifications/unread-count` | 헤더 벨 뱃지 |
| `POST /notifications/{id}/read` · `/read-all` | 읽음 처리 |

## web UX
- 제출: 비블로킹 → 토스트 "백그라운드에서 작성 중이에요. 완료되면 알림으로 알려드릴게요." 폼은 계속 사용 가능.
- 결과 패널: 해당 contentId가 있으면 상태 폴링(2~3s)으로 generating→done 전환 시 자동 렌더(없으면 안내).
- 헤더 벨: 범용 알림 피드로 확장. unread-count 폴링 뱃지, 드롭다운 목록, 클릭 시 링크 이동 + 읽음.
- 푸시 권한 미허용/미구독이어도 인앱 알림 + 목록으로 동작(푸시는 보조).

## 인수기준
1. 제출 즉시 응답(블로킹 없음) · 페이지를 떠나도 백그라운드 생성이 완료되어 목록에 done으로 남는다.
2. 완료 시 인앱 알림 1건 생성 + 헤더 벨 뱃지 증가, 클릭 시 해당 초안으로 이동·읽음 처리.
3. 푸시 구독이 있으면 완료 웹푸시 수신(없으면 인앱만, 에러 없음).
4. 실패 시 status='failed' + 실패 알림(사용자용 메시지), 재생성 가능.
5. 기존 동기 호출 경로/SPEC-007 회귀 없음(상태 컬럼 기본 'done' 호환).
6. 각 repo 검증 게이트 그린(api `./gradlew build` — detekt 포함, web `lint && tsc && test`).
